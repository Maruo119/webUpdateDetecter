【差分検知ロジック検証】
================================================================================

Lambda 関数の核となる差分検知ロジック（check_site() 関数）の動作を確認：

▶ 処理フロー

  1. State ロード
     └─ S3 から前回の href リストを取得
        {"AIG損保...": ["item2", "item3", "item4"]}

  2. 記事取得
     └─ サイト XPath で現在の記事を解析
        [{"href": "item1", "title": "..."}, ...]

  3. 差分検算
     └─ 新規 href 群から過去にない href を抽出
        prev_hrefs = {"item2", "item3", "item4"}
        curr_hrefs = {"item1", "item2", "item3", "item4"}
        diff_hrefs = {"item1"}  ← 差分

  4. Slack 判定
     └─ new_articles > 0 AND prev_hrefs > 0 の場合に通知
        ✓ パターン1: (1 > 0) AND (3 > 0) → SEND
        ✓ パターン2: (3 > 0) AND (1 > 0) → SEND

  5. State 更新
     └─ 現在の href リストを S3 に保存
        {"AIG損保...": ["item1", "item2", "item3", "item4"]}


▶ 初回実行時の動作

  初回実行時は prev_hrefs が空であり、Slack 通知は送信されません。
  代わりに、ベースラインとして全記事を state.json に記録します：

    [INFO] first run for AIG損保 ニュースリリース: recorded 4 item(s)

  これにより、初回実行による大量通知を防止しています。
